# 機能仕様 — メルマガ（属性別メール配信）

- **版**: 0.1（ドラフト）
- **最終更新**: 2026-06-22
- **位置づけ**: `docs/requirements.md`（要件定義書）と `CLAUDE.md` の確定判断を前提に、**メルマガ送信機能**を具体化する補足仕様。番号体系は本書独自に `FR-N*`（機能） / `SEC-N*`（セキュリティ）を用い、既存の `FR-*` / `SEC-*` を参照する。
- **矛盾時の優先**: `CLAUDE.md` > `requirements.md` > 本書。

---

## 1. 目的

大吉会計が、顧問先・見込み客・協業先の**担当者に対し、属性（配信トピック）でセグメントしたメールマガジンを配信**できるようにする。既存の顧客台帳（`companies` / `contacts`）の属性をそのままセグメント条件に再利用し、新たな名簿管理を二重に持たない。

達成したい状態:
- 担当者ごとに「配信同意」と「購読トピック」を管理できる。
- 配信トピック × 既存CRM属性（業種・エリア・規模・ステータス・needs/offers）の掛け合わせで宛先を絞り込める。
- 月額コストゼロ（C-1）と国内保管（C-7）を維持したまま配信できる。
- 特定電子メール法・個人情報保護法に沿った同意取得・配信停止を仕組みで担保する。

## 2. 確定した設計判断（本機能）

ユーザーと合意済み（2026-06-22）。実装中に迷ったらここを参照する。

- **N-1 配信基盤は「送信APIのみ＋自前構築」。** 外部の無料メール送信APIは**送信パイプとしてのみ**使う。配信トピック・購読リスト・配信履歴・配信停止はすべて Supabase 内に持つ。**顧客の個人情報を外部SaaSへ名簿同期しない**（C-7／OCRをTesseract.jsにした判断と同じ思想）。送信の瞬間だけ「宛先＋本文」をAPIへ渡す。
- **N-2 送信プロバイダはアダプタ化して差し替え可能にする。** 特定ベンダにロックしない。無料枠・日次上限・到達率で選定。
  - 無料枠は変動するため**実装時に公式で再確認**する（本プロジェクトの基本方針）。
  - 想定規模（N-3）なら無料枠に余裕。注意点は**日次送信上限**のみ（例: 日100通級のプロバイダでは300通＝2〜3日に自動分割、日300通級なら1日で完了）。
- **N-3 想定規模は 〜300通/回・月1〜2回。** 無料枠（月間）には十分収まる。設計は日次上限内へ自動スロットル送信できることを満たせばよい。
- **N-4 メルマガ属性（配信同意・購読トピック）は担当者（個人）単位で持つ。** 配信同意・配信停止は法的に個人単位が原則。1企業に複数担当者がいても各人で個別にON/OFF・トピック選択できる。
- **N-5 セグメント = 配信トピック（専用マスタ）× 既存CRM属性の組み合わせ。** 既存の絞り込み資産（`status`/`industry`/`area`/`size`/`needs`/`offers`）を再利用し、新規の名簿管理を作らない。
- **N-6 本文ドラフト生成は手元のClaude（MCP）に行わせる。** 外部LLM API課金を発生させない（FR-X3と同方針）。実送信は必ずUI（サーバー側DAL）から。

## 3. 機能要件（FR-N）

### 3.1 配信トピック・マスタ
- **FR-N1** admin が配信トピック（例: 税制改正ニュース／セミナー案内／経営お役立ち情報／年末調整／決算前リマインド）を追加・編集・無効化できる。`tags` と同じ「マスタ選択＋自由追加」作法（FR-T1/T2 に準拠）。

### 3.2 同意・購読管理（担当者単位 / N-4）
- **FR-N2** 担当者ごとに配信同意（opt-in）の有無と同意日時・取得経路を記録できる。
- **FR-N3** 担当者ごとに購読トピックを複数選択できる。
- **FR-N4** 担当者一覧/詳細から同意・購読状況を閲覧・編集でき、購読トピックや同意状況で絞り込める。
- **FR-N5** 配信停止（opt-out）された担当者は、以後すべての配信対象から自動的に除外される（抑制リスト）。
- **FR-N6**（任意）公開フォーム（`form-public.html`）に「メルマガ購読」チェックを追加し、確認メールで同意を確定（ダブルオプトイン）したうえで購読登録できる。

### 3.3 セグメント・配信作成
- **FR-N7** 配信トピック（FR-N1）と既存CRM属性（`status`/`industry`/`area`/`size`/`needs`/`offers`）をAND/ORで組み合わせて宛先セグメントを定義できる（N-5）。
- **FR-N8** セグメント定義に対する**対象人数を送信前に表示**する（同意なし・配信停止・メール無しは自動除外した実数）。
- **FR-N9** 件名・本文（簡易HTML＋プレーンテキスト）を作成し、差し込み変数（例 `{{氏名}}` `{{会社名}}`）を使える。
- **FR-N10** 自分宛などへの**テスト送信**ができる（本配信前の確認）。
- **FR-N11** 下書き保存、（拡張）予約配信ができる。
- **FR-N12** 本文ドラフトを手元のClaude（MCP）で生成し、スタッフが校正してから確定できる（N-6 / FR-X3）。

### 3.4 送信・履歴
- **FR-N13** 配信を実行すると、対象担当者を配信キュー（`newsletter_recipients`）に展開し、サーバー側ジョブが**日次上限内でスロットル送信**する（N-2/N-3）。実送信はサーバー側DALのみ（SEC-N3）。
- **FR-N14** 1配信の宛先ごとに送信結果（送信済/失敗/バウンス/配信停止スキップ）と送信日時・メッセージIDを記録する。
- **FR-N15** 配信一覧で過去の配信（下書き/予約/送信中/送信済/失敗）と送信数・（拡張）開封/クリックを確認できる。
- **FR-N16** 二重送信防止: 同一配信×同一担当者へは重複送信しない（`newsletter_recipients` で担保）。

### 3.5 配信停止（公開）
- **FR-N17** 各メール本文に、送信者名・住所・**配信停止リンク**を必ず含める（法令／§5）。
- **FR-N18** 配信停止リンク（認証不要・推測不能な `unsubscribe_token`）から、受信者が自分の配信を停止できる。停止は即座にDBへ記録され、以後の配信対象から除外される（FR-N5）。

## 4. データモデル（追加）

既存の作法に合わせる: 2層スキーマ（C-8）、論理削除（`deleted_at`）、`created_at`/`updated_at`、全テーブルRLS（SEC-1）。

### 4.1 newsletter_topics（配信トピック・マスタ）
`tags` と同じ思想（マスタ選択＋自由追加）。

| 列 | 型 | 説明 |
|---|---|---|
| id | uuid | 主キー |
| label | text（一意） | トピック名 |
| description | text | 説明（任意） |
| is_active | bool | 有効/無効 |
| sort | int | 並び順 |
| created_at | timestamptz | |

### 4.2 contacts への追加列（同意・配信停止 / N-4）
> 同意・配信停止は法令上「確実に引ける」必要があるため、`extra`(JSONB) ではなく**最初から列**にする（C-8 の例外運用）。

| 列 | 型 | 説明 |
|---|---|---|
| email_opt_in | bool | 配信同意フラグ |
| email_opt_in_at | timestamptz | 同意日時 |
| email_opt_in_source | text | 取得経路（手動/フォーム/CSV 等） |
| email_opt_out_at | timestamptz（null=有効） | 配信停止日時 |
| unsubscribe_token | uuid | 配信停止リンク用（推測不能・一意） |

### 4.3 contact_subscriptions（担当者 × 購読トピック）
| 列 | 型 | 説明 |
|---|---|---|
| id | uuid | 主キー |
| contact_id | FK→contacts | |
| topic_id | FK→newsletter_topics | |
| status | enum | `購読` / `停止` |
| source | text | 登録経路 |
| created_at | timestamptz | |

（`contact_id` × `topic_id` は一意）

### 4.4 newsletters（配信＝キャンペーン）
| 列 | 型 | 説明 |
|---|---|---|
| id | uuid | 主キー |
| subject | text | 件名 |
| body_html | text | 本文（HTML） |
| body_text | text | 本文（プレーン） |
| from_name | text | 送信者名 |
| reply_to | text | 返信先 |
| segment_def | jsonb | 使った絞り込み条件（再現・監査用 / N-5） |
| target_topic_ids | uuid[] | 対象トピック |
| status | enum | `下書き`/`予約`/`送信中`/`送信済`/`失敗` |
| scheduled_at | timestamptz | 予約時刻（任意） |
| sent_at | timestamptz | 送信完了時刻 |
| created_by | uuid（→auth.users） | 作成者（監査） |
| created_at / updated_at | timestamptz | |

### 4.5 newsletter_recipients（配信明細・履歴）
| 列 | 型 | 説明 |
|---|---|---|
| id | uuid | 主キー |
| newsletter_id | FK→newsletters | |
| contact_id | FK→contacts | |
| email_snapshot | text | 送信時点の宛先（後で変わっても履歴保持） |
| status | enum | `待機`/`送信済`/`失敗`/`バウンス`/`停止スキップ` |
| error | text | 失敗理由 |
| message_id | text | プロバイダのメッセージID |
| sent_at | timestamptz | |

> **2層スキーマの当て方（C-8）**: 試験的・低頻度の配信用項目は、まず `contacts.extra` / `newsletters.segment_def` に入れて運用 → 絞り込みで多用されると分かったら正式テーブル/列へ昇格。ただし**同意・配信停止（§4.2）だけは最初から列**（法令対応）。

## 5. 法令対応（特定電子メール法・個人情報保護法）

会計事務所＝信頼が要のため、機能の一部として最初から組み込む。

- **L-1 オプトイン（事前同意）**: 営業メルマガは原則同意が必要。同意日時・経路を記録（§4.2）。
- **L-2 表示義務**: 送信者名・住所・配信停止方法を本文に必ず記載（FR-N17）。
- **L-3 配信停止の確実な履行**: ワンクリック停止 → opt-out 記録 → 以後の配信から自動除外（FR-N5/N18）。
- **L-4 ダブルオプトイン（推奨）**: 公開フォーム経由の購読は確認メールで同意確定（FR-N6）。
- **L-5 バウンス/苦情の自動抑制（拡張）**: 送信プロバイダのWebhookでバウンス・苦情を受け取り、自動で配信停止フラグを立て到達率を守る。

## 6. セキュリティ要件（SEC-N / 既存方針との整合）

- **SEC-N1** 新テーブル（`newsletter_topics`/`contact_subscriptions`/`newsletters`/`newsletter_recipients`）すべてで RLS 有効化。`authenticated` のみ許可、`anon` 不可（SEC-1/SEC-2 に準拠）。
- **SEC-N2** 配信停止の公開ページは `unsubscribe_token` ベースの限定操作のみ（opt-out フラグ反転だけ。他データは読めない）。レート制限を設け、`form-public` と同じく公開だが書き込みを最小化する。
- **SEC-N3** 送信プロバイダのAPIキーは**サーバー側のシークレットのみ**（Cloudflare/Supabase のシークレット機構）。`NEXT_PUBLIC_*` に置かない（SEC-5）。一斉送信は必ずサーバー側DAL経由で、anonクライアントから直接送信させない（SEC-4/SEC-7）。
- **SEC-N4** 購読者CSVの取込/書き出しは既存の検証＋フォーミュラインジェクション対策を流用（SEC-6 / FR-S2/S4）。
- **SEC-N5** 到達率・なりすまし対策として、事務所の独自ドメインに SPF / DKIM / DMARC を設定して送信する。

## 7. 配信パイプライン

```
スタッフ → メルマガ作成（件名・本文・セグメント条件）
        → 対象人数プレビュー（同意なし/停止/メール無しを除外）
        → 確定 → 配信キュー（newsletter_recipients を展開）
        → サーバー側ジョブ（Cloudflare Workers Cron Triggers〔無料〕
            または Supabase scheduled Edge Function）が日次上限内でスロットル送信
        → 送信API（アダプタ経由 / N-2）→ 結果を明細に記録
受信者 → メール内「配信停止」リンク（公開・token）→ opt-out をDBへ
```

- 送信プロバイダはアダプタインターフェース（`send({to, subject, html, text})`）の裏に隠し、無料枠・日次上限で差し替え可能にする（N-2）。
- 定期実行は無料の Cloudflare Workers Cron か Supabase の scheduled function を用いる。

## 8. 画面（追加）

`mockups/assets/app.js` のナビ構成に合わせ、`連携・収集` グループへ `メルマガ` を追加。属性マスタは `管理` 配下。

| 画面 | 内容 |
|---|---|
| メルマガ一覧 | 過去の配信（状態・送信数・〔拡張〕開封/クリック） |
| メルマガ作成 | 件名・本文（簡易HTML＋差し込み）・セグメント選択・対象人数・テスト送信・下書き/予約 |
| 配信結果 | 1配信の宛先ごとの送信/失敗/停止スキップ明細 |
| 購読者管理 | 担当者の同意・購読トピックの一覧/絞り込み（既存の担当者画面に統合も可） |
| 管理 > メルマガ属性マスタ | 配信トピックの追加・統廃合（`admin-masters` に追加） |
| （公開）配信停止ページ | 認証不要・token で opt-out（`form-public.html` の系譜） |

横断状態（空/読込中/エラー/403/404）は FR-U2 に準拠。

| フェーズ | 内容 |

| **MVP** | 配信トピック・マスタ、担当者の同意/購読（N-4）、簡易HTML本文＋差し込み、セグメント（トピック×CRM属性）＋対象人数表示、テスト送信、スロットル送信、配信履歴、公開・配信停止ページ、ドメイン認証 |

## 10. 未確定事項（次に決める）

1. **送信プロバイダの第一候補**: 無料枠・日次上限・到達率を実装時に比較して確定（N-2）。300通/回を1日で送るか、数日に分割するかも併せて決める。
2. **送信元ドメイン**: 大吉会計の独自ドメイン（SPF/DKIM/DMARC設定先）。
3. **送信者表示情報**: 本文フッタに載せる事務所名・住所・問い合わせ先（L-2）。
4. **本文形式の範囲**: MVPは簡易HTML＋差し込みでよいか（リッチエディタは拡張）。
5. **購読者管理の置き場所**: 担当者画面に統合するか、独立画面にするか。
