## 1. 目的とゴール

大吉会計（会計事務所）が、顧問先・見込み客を一元管理し、**顧客同士のビジネス紹介（協業先紹介・顧客紹介）を仕組み化**するためのクラウドCRM。

達成したい状態:

- 顧客（企業・個人事業主）の情報を1か所で管理し、条件で素早く絞り込める。
- 「この顧客が求めていること（needs）」と「別の顧客が提供できること（offers）」を突き合わせ、紹介候補を自動提示できる。
- 実施した紹介を履歴として残し、成果（成立/不成立）まで追える。
- ブラウザUIからも、手元の Claude（MCP経由）からも**同じデータ・同じ結果**で操作できる。
- 月額コストゼロを維持する。

## 2. 用語定義

| 用語 | 定義 |
|---|---|
| 顧客（customer） | 大吉会計の顧問先・見込み客。企業または個人事業主。 |
| needs | その顧客が**求めていること**（例: 集客、仕入先、資金調達）。タグで表現。 |
| offers | その顧客が**提供できること**（例: Web広告、食材卸、税務相談）。タグで表現。 |
| マッチング | ある顧客の needs と別の顧客の offers のタグの重なりで相性をスコア化し、紹介候補を出すこと。 |
| 協業先紹介 | A社の needs を B社の offers が満たす紹介（A に B を紹介）。 |
| 顧客紹介 | A社の offers を B社の needs が必要としている紹介（B に A を紹介）。※方向が逆なだけで同一ロジック。 |
| 紹介（referral） | 実際に行った/行う予定の紹介の記録。提案→打診→成立/不成立を追跡。 |
| スタッフ | 大吉会計の職員。本システムのログインユーザー。 |

## 3. 前提・制約（CLAUDE.md より／交渉不可）

- **C-1 無料が最優先。** 月額コストゼロを維持。スタック選定はこれに従う。
- **C-2 スタック確定。** Next.js (App Router/TS) + OpenNext on Cloudflare + Supabase (Tokyo)。
- **C-3 認証は Supabase Auth**（メール+パスワード）。自前実装しない。
- **C-4 全テーブル RLS 有効化。** `auth.uid()` 起点。`service_role` キーはサーバー側のみ、ブラウザに出さない。
- **C-5 認可は DAL（データアクセス層）で必ず実施。** ミドルウェア（Next.js 16 では `proxy.ts`）だけに頼らない（CVE-2025-29927 の教訓）。
- **C-6 マッチングのスコアリングは Postgres 側（SQL/RPC）で完結。** 外部LLM APIに課金しない。
- **C-7 機微情報を扱う。** Supabase は Tokyo リージョンで国内保管。
- **C-8 スキーマ進化前提（2層構造）。** コア項目は列、試験的項目は `extra`(JSONB)、多用されたら列に昇格。

## 4. スコープとフェーズ

MVP は**顧客台帳（企業＋担当者＋名刺画像）の完成**（決定事項）。マッチングと紹介履歴はフェーズ2に回すが、データモデルは最初からフェーズ2を見据えて設計する。

| フェーズ | 内容 | 状態 |
|---|---|---|
| **Phase 0 — 基盤** | プロジェクト初期化、Supabase接続、Auth、Storageバケット、RLS雛形、CI/型生成、Cloudflareデプロイ確認 | 実装済み（本番デプロイ待ち） |
| **Phase 1 — 顧客台帳（MVP）** | 企業＋担当者のCRUD、検索・絞り込み、タグ（マスタ+自由）、CSV取込/書き出し、**名刺画像（表/裏）保存**、担当者の手動登録 | 実装済み |
| **Phase 1.5 — 名刺OCR** | 名刺画像から項目を自動抽出→確認・補正して企業＋担当者を作成（抽出エンジンは §6.5.1 で決定） | 実装済み（Tesseract.js） |
| **Phase 2 — マッチング & 紹介** | needs↔offers のスコアリング(RPC)、紹介候補提示、紹介履歴の記録・追跡 | 実装済み |
| **Phase 3 — Claude連携の磨き込み** | MCPからの同一ロジック利用、紹介提案文の生成（手元Claude） | 実装済み（MCP本番RPC対応） |

**MVP の完了条件（Definition of Done）**: スタッフがログインし、企業を登録・編集・論理削除・絞り込みでき、企業配下に担当者を追加して**名刺画像（表/裏）を保存・閲覧**でき、CSVで企業を一括取込/書き出しでき、全操作がRLSで保護され、名刺画像が署名URLでのみ閲覧できる状態。

> **Phase 2–3 の追加機能**: メルマガ配信（`docs/newsletter-feature.md`）と 期限・タスク（スケジュール管理, §6.10）は MVP 後の追加機能。ただしデータモデルは本書（§8.7 ほか）で先に定義しておき、後付けの破壊的変更を避ける。**いずれもアプリ層は実装済み**（メルマガ＝Resend 実送信・購読解除トークン／期限・タスク＝自動生成 RPC＋4ビュー＋iCal 書き出し）。

## 5. ユーザーとロール

権限は**全員で共有**（決定事項）。当面ロールは2種類のみ。

| ロール | 説明 | 権限 |
|---|---|---|
| staff（一般スタッフ） | 大吉会計の職員 | 全顧客の閲覧・登録・編集・削除、CSV取込/書き出し、マッチング利用 |
| admin（管理者） | 事務所の管理担当 | staff の全権限 ＋ ユーザー招待/無効化、タグマスタの管理 |

- **アカウント発行は招待制**（サインアップは一般開放しない）。当面は管理者が Supabase で発行 or 簡易招待。【要確認: 6.1 参照】
- 「全員で共有」のため、顧客レコードに担当者ベースのアクセス制御は**かけない**。ただし**社内**の担当スタッフを示す任意項目 `owner` は持ってよい（表示・絞り込み用、アクセス制御には使わない）。これは名刺由来の**先方担当者**（`contacts`）とは別概念。

## 6. 機能要件（FR）

### 6.1 認証・アカウント
- **FR-A1** メール+パスワードでログイン/ログアウトできる（Supabase Auth）。
- **FR-A2** 未ログインでは顧客データに一切アクセスできない（RLSで強制、UIでも遮断）。
- **FR-A3** パスワードリセット（メール）ができる。
- **FR-A4** 管理者がスタッフを招待/無効化できる。【MVP範囲か要確認 — 当面はSupabaseダッシュボード運用で可】
- **FR-A5**（G-01）全画面共通のユーザーメニューからログアウトできる。
- **FR-A6**（G-01）自分のアカウントを管理できる（表示名、メール確認、パスワード変更）。S-14。

### 6.2 顧客（企業）台帳（Phase 1 / MVP）

> 顧客＝企業 `companies`。先方の担当者・名刺は §6.5（別テーブル）。社内担当 `owner` は別概念。

- **FR-C1** 企業（顧客）を新規登録できる（必須: `type`, `name`。他は任意）。
- **FR-C2** 顧客一覧を表示できる（ページング/並び替え対応）。
- **FR-C3** 顧客詳細を表示できる。
- **FR-C4** 顧客を編集できる。
- **FR-C5** 顧客を**論理削除**できる。`deleted_at` を立てるのみで物理削除しない。一覧・検索・マッチングの既定では除外し、必要に応じて復元できる（監査・誤削除対策）。物理削除は admin の例外操作とし、当面は提供しない。
- **FR-C6** 次の条件で絞り込み・検索できる: フリーワード（name/notes）、`type`、`industry`、`area`、`status`、`needs` タグ、`offers` タグ。複数条件のAND絞り込み。
- **FR-C7** `extra`(JSONB) に試験的項目をマイグレーションなしで追加・表示・編集できる（少なくとも詳細画面で閲覧/編集）。
- **FR-C8**（G-03）論理削除した企業/担当者を**ゴミ箱で閲覧・復元**できる（S-09）。完全削除は admin の例外操作。
- **FR-C9**（G-04）登録・CSV取込・名刺スキャン時に**重複企業を検出**し（社名/メール等の近似）、既存企業への担当者追加 or マージを選択できる。
- **FR-C10**（G-12, 任意）よく使う絞り込み条件を保存/URL共有できる。

### 6.3 タグ管理（Phase 1）
- **FR-T1** needs/offers のタグは**マスタから選択**できる（決定事項）。
- **FR-T2** マスタに無いタグは**その場で自由追加**でき、追加後はマスタに蓄積される（決定事項）。
- **FR-T3** needs と offers は**同一のタグ語彙**を共有する（例「Web広告」は誰かの offers であり別の誰かの needs になりうる）。マッチングを成立させるための前提。
- **FR-T4** 管理者がタグの統廃合（表記ゆれの名寄せ・別名統合）を行える。【MVPでは簡易、本格運用はPhase2以降】
- **FR-T5**（G-11）タグ入力はマスタからのオートコンプリート補完、未知タグは**確認の上で追加**する（表記ゆれ防止＝マッチング精度に直結）。

### 6.4 CSV（Phase 1）
- **FR-S1** CSVから顧客を一括取込できる。列構成は `companies`（企業）スキーマに対応。**CSVは企業単位**で、担当者・名刺は名刺スキャン（§6.5）または担当者画面から登録する（担当者のCSV一括取込は §12 の検討事項）。
- **FR-S2** 取込時にスキーマ検証・型チェックを行い、不正値・列ずれを**投入前に弾く**（エラー行を提示）。
- **FR-S3** 顧客一覧（絞り込み結果を含む）をCSVで書き出せる。
- **FR-S4** 書き出し時に**CSVフォーミュラインジェクション対策**を行う（`=` `+` `-` `@` で始まるセルをエスケープ）。
- **FR-S5** タグ列（needs/offers）は区切り文字 `;`（セミコロン）で複数値を表現し、取込時にタグマスタへ反映する。
- **FR-S6**（G-09）取込用テンプレCSV（ヘッダ付き・空行）をダウンロードできる。
- **FR-S7**（G-09）取込は**検証OKの行のみ投入し、NG行はスキップして理由付きで一覧表示**する（既定方針）。
- **FR-S8**（G-10, 後続/任意）担当者のCSV一括取込（所属企業との対応付け）を将来サポートする。

#### CSVフォーマット（確定 — 一番シンプル）

既存CSVは無いため、ゼロから最小構成で定義する。

- **文字コード**: UTF-8（BOM付き。Excelでの文字化け防止）
- **区切り**: カンマ。タグ（求めてること/提供できること）内の複数値は `;` 区切り。
- **ヘッダ行**: 必須（日本語ヘッダ。スタッフが直接編集できるように）。
- **取込列（この順・ヘッダ名固定）**:

  | ヘッダ | 対応列 | 必須 | 例 |
  |---|---|---|---|
  | 種別 | type | ✓ | 企業 |
  | 名称 | name | ✓ | 株式会社サンプル |
  | 業種 | industry |  | 飲食 |
  | エリア | area |  | 東京都 |
  | 規模 | size |  | 1千万〜5千万 |
  | 求めてること | needs |  | 集客;資金調達 |
  | 提供できること | offers |  | 食材卸 |
  | ステータス | status |  | 顧問中 |
  | メモ | notes |  | 紹介経由 |

- **取込ルール**: `種別`/`名称` 以外は空欄可。`種別`・`ステータス`・`業種`・`エリア`・`規模` はマスタ/区分に存在する値か検証（表記ゆれは弾く or 候補提示）。`求めてること`/`提供できること` の未知タグはマスタへ追加。検証NG行は取込せずエラー一覧を提示（FR-S2）。
- **書き出し列**: 取込列に加え、先頭に `ID`、末尾に `登録日` を付ける（往復しやすいよう取込時は `ID`/`登録日` を無視）。`deleted_at` がある行（論理削除済み）は既定で書き出さない。
- **安全対策**: 書き出し時、セル先頭が `=` `+` `-` `@` の値は先頭に `'` を付けてエスケープ（FR-S4 / SEC-6）。

### 6.5 担当者・名刺・OCR（Phase 1 / 一部 1.5）

先方の担当者（名刺の人物）は企業と分離し、**1企業 : 複数担当者**で持つ（`contacts`）。

- **FR-P1**（P1）1つの企業に複数の担当者を登録・編集・論理削除できる（氏名・フリガナ・役職・部署・電話・携帯・メール等）。
- **FR-P2**（P1）担当者ごとに**名刺画像を表・裏それぞれ**アップロード・保存・閲覧できる（Supabase Storage の非公開バケット）。
- **FR-P3**（P1.5）名刺画像から**OCRで項目を自動抽出**できる（会社名/氏名/フリガナ/役職/部署/電話/FAX/メール/住所/URL 等）。抽出エンジンは §6.5.1 で決定。
- **FR-P4**（P1.5）OCR結果は**確定前に人が確認・補正**してから保存する（誤抽出をそのまま登録しない）。
- **FR-P5**（P1→P1.5）**名刺起点の顧客作成フロー**: 名刺撮影/アップロード → （OCR抽出）→ 確認・補正 → 新規企業＋担当者として登録。既存企業がヒットすれば担当者だけ追加する。
- **FR-P6**（P1）名刺画像は機微情報（個人情報）として扱い、非公開バケット＋**署名URL**でのみ閲覧（§9 SEC-9）。
- **FR-P7**（P1）企業を論理削除したら、配下の担当者・名刺も既定で非表示にする。
- **FR-P8**（P1, G-05）名刺画像を差し替え・削除でき、ビューアで**表/裏切替・拡大**表示できる。
- **FR-P9**（P1.5, G-07）OCR前に画像の**トリミング/回転/明るさ調整**ができる（精度向上）。
- **FR-P10**（P1.5, G-08）OCRの**低信頼フィールドをハイライト**し、確認を促してから保存する。
- **FR-P11**（任意, G-06）1担当者に複数の名刺（履歴）を保持・表示できる（転職・名刺更新）。

#### 6.5.1 OCRエンジン（確定: ブラウザ内 Tesseract.js）

- **方式**: ブラウザ内（クライアントサイド）OCR = **Tesseract.js**（日本語データ `jpn`）。
- **理由**: 完全無料（C-1）＋ 画像をブラウザ外の外部OCRサービスに送らない（C-7に最適、SEC-10をクリア。画像は自前の Supabase Tokyo Storage にのみ保存）。
- **精度対策**: 日本語名刺の素のOCR精度は中〜弱のため、
  1. 前処理（グレースケール/二値化/傾き補正）で認識率を上げる、
  2. 抽出テキストから会社名/氏名/役職/部署/電話/FAX/メール/URL/住所への**項目振り分けは自前ヒューリスティクス**で行う（例: メール=`@`、電話=`0\d{1,4}[-(]`、URL=`https?://`、郵便番号=`\d{3}-\d{4}`）、
  3. **必ず人が確認・補正してから保存**（FR-P4）。粗い抽出はこの確認ステップで吸収する。
- **保存との独立**: 画像保存（FR-P2）はOCRの成否に依存しない。OCRの生結果は `business_cards.ocr_raw` に残す。
- **将来の拡張余地**: 精度が不足する場合の代替として、手元Claude(MCP)での高精度抽出、またはサーバー側OCRを検討（要件は据え置き）。

### 6.6 マッチング（Phase 2）
- **FR-M1** ある顧客を起点に、相性の高い紹介候補を**スコア順**で提示する。
- **FR-M2** スコアリングは Postgres 側（SQL/RPC関数）で完結し、UI と MCP(Claude) が同一関数を呼ぶ（C-6）。
- **FR-M3** スコアの基本定義（v1）: `score = |A.needs ∩ B.offers| + |A.offers ∩ B.needs|`（タグの重なり数）。同一顧客同士・自己マッチは除外。
- **FR-M4** 候補表示時に「どのタグが重なったか（協業先紹介/顧客紹介のどちら方向か）」を根拠として併記する。
- **FR-M5** マッチング精度のため、`industry`・`area`・`size` は**構造化（マスタ/区分選択）**して保持する（自由入力にしない）。これにより同一/近接エリア・関連業種での重み付けや絞り込みが効く。スコアv1はタグ重なり数のみ、エリア/業種の重み付け・しきい値足切りは v2 の拡張とする。
- **FR-M6**（G-17, P2拡張）起点指定なしの「おすすめマッチング」一覧/ダイジェストを提示できる。

### 6.7 紹介履歴（Phase 2）
- **FR-R1** 紹介を記録できる: `from_customer`（紹介する側 or 起点）、`to_customer`、`kind`（協業先紹介/顧客紹介）、根拠タグ、`status`、メモ。
- **FR-R2** 紹介のステータスを更新できる: `提案` → `打診中` → `成立` / `不成立`。
- **FR-R3** 顧客詳細から、その顧客が関係する紹介履歴（紹介した/された）を一覧できる。
- **FR-R4** マッチング候補から1クリックで紹介レコードを起票できる（候補→記録への導線）。
- **FR-R5**（G-18, 任意/P2）打診中など未完了の紹介にフォロー/リマインドを出せる。

### 6.8 Claude/MCP 連携（Phase 3）

- **FR-X2** MCP は **read-only モード＋対象プロジェクト限定**で接続する（CLAUDE.md 開発環境方針）。
- **FR-X3** 紹介提案文・要約の文章生成は**手元の Claude（MCP経由）**に行わせ、外部LLM API課金を発生させない（C-6）。

### 6.9 横断UI・ナビ・状態（Phase 1）

- **FR-U2**（G-19）全画面で **空 / 読込中 / エラー / 403（権限なし）/ 404** の標準状態を用意する。

### 6.10 期限・タスク（スケジュール管理）（Phase 2–3）

会計事務所の業務サイクル（決算・申告・納付・年末調整）と社内の手動タスクを**1つの期限ハブ**で管理する。期限は各企業の**決算月・課税区分から自動生成**し、面談フォローや資料回収などは**手動タスク**として足す。生成ロジックはマッチング同様 DB 側（SQL/RPC）に集約し、UI と MCP(Claude) が同一結果を得る（C-6 の思想）。対応する画面は S-15（設計モック `mockups/schedule.html`）。

- **FR-D1** 企業ごとに**決算月**（`fiscal_month` 1–12）と課税区分（消費税: 課税/免税/簡易、源泉所得税の納期特例の有無）を保持し、税務期限を自動生成する。
- **FR-D2** 自動生成する期限の種別（最小セット。法令準拠の最終確認は実装時）:
  - 決算準備リマインド（決算月の約3ヶ月前）
  - 法人税・消費税の**申告期限**（決算日の2ヶ月後）
  - 消費税の**中間申告・納付**（課税区分・前年税額に応じて）
  - **年末調整・法定調書**（12月〜翌1月）、**償却資産申告**（1/31）
  - **源泉所得税の納付**（原則毎月10日 / 納期特例は 1/20・7/10）
- **FR-D3** **手動タスク**（面談フォロー・資料回収・期日のある社内ToDo）を企業に紐付けて追加・編集・完了・論理削除でき、社内担当（`owner`/スタッフ）を割り当てられる。
- **FR-D4** 期限・タスクを横断一覧で表示し、**遅延 / 今週 / 今月 / 来月以降**にグルーピング、**種別・担当・状態**で絞り込める。
- **FR-D5** 状態は `未対応` → `対応中` → `完了`。期日超過かつ未完了は**遅延**として最上部に強調表示し、放置を防ぐ。
- **FR-D6** **月カレンダー表示**（予定のある日にマーカー、種別で色分け）と一覧表示を切り替えられる。
- **FR-D7** 自動生成の期限は原則編集不可（元データ＝決算月等の変更で**冪等に再計算**）。完了・スヌーズは可。手動タスクのみ内容を自由編集できる。
- **FR-D8** 企業詳細（S-03）に「**この企業の期限・タスク**」ミニ一覧を表示する（§6.2 と連動）。
- **FR-D9** ダッシュボード（S-01）の「対応が必要」に、遅延・今週の期限・未完了タスク件数を表示する（FR-U1 と連動）。
- **FR-D10**（連携）**Googleカレンダーへ書き出し**できる。打ち合わせ（`meetings`）と同じカレンダー連携基盤を用いる。OAuthトークンはサーバー側保管（SEC-3/5）、外部へ送るのは予定タイトル等に限定し機微情報を出さない（C-7）。
- **FR-D11** 生成ロジックは DB 側（SQL/RPC）に集約し、MCP からは read-only で「今週の期限」「遅延一覧」等を問い合わせできる（FR-X1 と同一思想・将来）。
- **FR-D12**（任意/P3）期日のN日前・前営業日のリマインド通知（メール/アプリ内）。FR-R5（紹介リマインド）と通知基盤を共有する。

### 6.11 企業の資料・ファイル（ドキュメント保管）（Phase 1.5–2）

企業ごとに**説明資料・商品カタログ・契約書・決算書・提案資料など**のファイル（PDF/画像/Excel/Word 等）を保管する。名刺画像と同じ**非公開バケット＋署名URL**方式で機微情報として扱う（SEC-9/SEC-12）。対応画面は企業詳細（S-03）の「資料」タブ（設計モック `mockups/company-detail.html`）。

- **FR-F1** 企業ごとに資料ファイルを複数アップロード・一覧・プレビュー・ダウンロード・削除できる。
- **FR-F2** ファイルに**種別（カテゴリ）**を付与できる（`契約書` / `決算書` / `商品・サービス資料` / `提案資料` / `その他`。カテゴリは admin が調整可）。種別で絞り込める。
- **FR-F3** ファイル実体は **Supabase Storage の非公開バケット**に保存し、テーブルにはパス＋メタ（ファイル名・サイズ・MIME）のみ持つ。閲覧・DLは**有効期限付き署名URL**でのみ（直URL公開はしない。名刺＝SEC-9 と同一方式）。
- **FR-F4** アップロード時に**拡張子・MIMEタイプ・サイズ**を検証する（許可リスト方式: pdf/jpg/png/webp/xlsx/docx 等、1ファイル上限の目安 20MB）。実行ファイル等は弾く（NFR-10 / SEC-12）。
- **FR-F5** 企業詳細（S-03）の「**資料**」タブで一覧・カテゴリ絞り込み・プレビュー（PDF/画像）を行う（§6.2 と連動）。
- **FR-F6** 企業を論理削除したら配下の資料も既定で非表示にする（名刺 FR-P7 と同様）。
- **FR-F7** 削除は**論理削除**（`deleted_at`）。Storage実体の物理削除は admin の例外操作（誤削除・参照切れ対策）。
- **FR-F8** ストレージは**無料枠（Supabase Storage 約1GB）**を前提に、企業/全体の使用量を把握でき、上限接近時に警告する（C-1 無料維持）。大容量・大量保管が必要になれば外部保管の方針を別途検討（§12）。
- **FR-F9**（任意）アップロード/閲覧/DL/削除を**活動履歴・監査ログ**に記録する（§6.10 活動・NFR-9 と連携）。
- **FR-F10** 全社横断の資料検索を**提供する（本採用）**。ファイル名・カテゴリ・会社・登録者で横断検索し、メタデータ（ファイル名・種別・会社・サイズ・登録者・登録日）の一覧を返す。詳細は §6.11.1。
- **FR-F11** 資料検索のロジックは DB 側（SQL/RPC）に集約し、ブラウザUI（`mockups/documents.html`）と MCP（`search_documents`）が**同一結果**を返す（C-6 / FR-X1）。
- **FR-F12** 検索結果の会社名から企業詳細（S-03）の「資料」タブへ遷移できる。並び替えは登録の新しい順を既定とする。
- **FR-F13** Claude(MCP) には**メタデータのみ**返し、ファイル本体・署名URLは返さない（SEC-12 / SEC-X4）。実体の閲覧・DLは UI 側で有効期限つき署名URLを都度発行する。

#### 6.11.1 資料の全社横断検索（採用 — FR-F10〜F13）

企業詳細の「資料」タブ（§6.11）が**1社単位**なのに対し、全社の保管資料を**横断**で探す画面（`mockups/documents.html`、左ナビ「資料」）。「あのカタログ、どの会社だったか」を即解決し、容量監視（FR-F8）も兼ねる。

- **検索条件**: フリーワード（ファイル名・カテゴリ・社名の部分一致）、カテゴリ、会社、登録者。結果は登録の新しい順。
- **結果列**: 会社 / ファイル名 / 種別 / サイズ / 登録者 / 登録日。会社 → 企業詳細（S-03 資料タブ）へリンク。
- **同一ロジック（C-6）**: UI も MCP `search_documents` も同じ DB側RPC を呼ぶ（二重メンテ回避）。MCP はメタのみ返し（SEC-12/SEC-X4）、実体閲覧は UI の署名URL（FR-F3）。
- **容量監視**: 全社・企業別の合計サイズを表示し、無料枠（約1GB）接近で警告（FR-F8）。
- 実装状況: デモ MCP `mcp/crm-demo/`（`search_documents`・メタのみ）＋ UI モック `mockups/documents.html` を実装済。本番は RPC へ差し替え（ツール仕様は `docs/claude-integration.md` §4）。

### 6.12 活動履歴（タイムライン）（Phase 1.5–2）

顧客との**すべての接点**（電話・面談・メール・議事録・紹介・メルマガ・フォーム・名刺・タスク・メモ）を時系列の**統一フィード**で横断表示する。多くは各画面の操作・連携から**自動記録**し、電話・訪問・メモは**手動追加**する。「最近この顧客に何をしたか」を即把握し、フォロー漏れ・休眠の兆候を防ぐ。**監査ログ（NFR-9）を兼ねる**。対応画面は活動履歴（`mockups/activities.html`）＋企業詳細（S-03）のミニ・タイムライン。要件番号は活動=ACtivity で **FR-AC***。

- **FR-AC1** 活動を時系列の統一フィードで横断表示する。種別: `電話` / `面談・訪問` / `メール` / `議事録` / `紹介` / `メルマガ` / `フォーム` / `名刺` / `資料` / `タスク` / `メモ`（種別は admin が調整可）。
- **FR-AC2** 活動は各機能の操作・連携から**自動記録**する: 議事録の取込（Notta連携）、紹介の起票・打診（FR-R）、メルマガ送信（FR-N）、フォーム受信・取込、名刺OCR取込（FR-P）、期限・タスクの完了（FR-D5）、資料のアップロード/閲覧・DL（FR-F9）。
- **FR-AC3** 電話・訪問・メモ等は「＋活動を記録」で**手動追加**できる（種別・企業・内容・担当・日時）。手動活動は編集・論理削除できる（自動記録は原則編集不可）。
- **FR-AC4** 種別・担当・期間（今日/今週/今月）・企業で絞り込み、フリーワード（内容・企業）で検索できる。
- **FR-AC5** 各活動から**元レコードへ遷移**できる（議事録→議事録詳細、紹介→紹介、メルマガ→配信結果、フォーム→受信箱、名刺→担当者 等）。遷移先が論理削除済みなら非活性化する。
- **FR-AC6** 企業詳細（S-03）に「**この企業のタイムライン**」ミニ一覧を表示する（FR-D8 と同じ右カラム方針・§6.2 連動）。
- **FR-AC7** 「**フォロー待ち**」（未対応の接点＝未取込フォーム・未フォローの紹介・未完了の手動活動など）を集計し、活動画面とダッシュボード（FR-U1 / FR-D9）に表示してフォロー漏れを防ぐ。
- **FR-AC8** 自動記録には **actor（操作者）と occurred_at（発生時刻）** を残し、削除・取込・紹介起票・配信・資料DL 等の重要操作を活動として記録する（監査ログ＝NFR-9 を兼ねる）。
- **FR-AC9** 自動記録は**冪等**とし、同一ソースイベントを重複記録しない（`source_kind` ＋ `source_id` で一意）。
- **FR-AC10**（将来/MCP）「最近この顧客に何をしたか」を MCP から read-only で問い合わせできる（`get_company_timeline` 等・FR-X1 と同思想。`docs/claude-integration.md` §4 候補）。
- **FR-AC11**（任意）古い自動活動の保持期間・アーカイブ方針（無料枠維持 C-1）は将来検討（§12）。

## 7. 非機能要件（NFR）

- **NFR-1 コスト**: 無料枠内で運用（Cloudflare Workers/OpenNext, Supabase 無料プラン, Tokyoリージョン）。
- **NFR-2 セキュリティ**: 第9章の要件をすべて満たす（最優先）。
- **NFR-3 可用性/規模**: 想定ユーザーは事務所の少人数（〜数十名）、顧客件数は数百〜数千件規模を想定。無料枠で十分。
- **NFR-4 性能**: 一覧/絞り込みは数千件で体感1秒以内。検索対象列・タグにインデックスを張る。
- **NFR-5 型安全**: TypeScript型は Supabase スキーマから自動生成し、DBとUIの型を一致させる。
- **NFR-6 保守性/スキーマ進化**: 変更は `supabase/migrations/` のSQLで管理。コア列 vs `extra`(JSONB) の2層構造（C-8）。
- **NFR-7 監査性**: `created_at` / `updated_at` を全主要テーブルに持つ。削除は `deleted_at` による論理削除（採用済み）。
- **NFR-8 国内保管**: Supabase は Tokyo リージョン固定（C-7）。
- **NFR-9 監査ログ**（G-16, 任意）: 削除・取込・紹介起票などの重要操作の操作ログ（誰が・いつ）を残す。Phase 2 以降で検討。
- **NFR-10 入力検証**（G-14）: 必須・メール・電話・郵便番号・URL 等の検証規約を、フォーム/CSV/名刺OCR 共通で適用する。
- **NFR-11 レスポンシブ**（G-20）: モバイル/タブレット/デスクトップに対応。名刺撮影はモバイルのカメラ起動に対応する。

## 9. セキュリティ要件（最優先 — CLAUDE.md 反映）

- **SEC-1** 全テーブルで RLS 有効化。`companies`/`contacts`/`business_cards`/`tags`/`referrals` すべてにポリシーを設定。
- **SEC-2** ポリシー方針（全員で共有）: `authenticated` ロールは SELECT/INSERT/UPDATE/DELETE 可、`anon` は不可。`auth.uid()` 起点で組む。`referrals.created_by` は起票者を記録（監査）。
- **SEC-3** `service_role`/`sb_secret_*` キーはサーバー側（Workers/サーバーコンポーネント）のみ。ブラウザには anon/publishable のみ。
- **SEC-4** 認可は DAL で必ず実施（C-5）。ミドルウェア（`proxy.ts`）単独に依存しない。
- **SEC-5** シークレットはコードに置かない。`.env.local`（gitignore）/ Cloudflare・Supabase のシークレット機構。`NEXT_PUBLIC_*` に機微情報を入れない。
- **SEC-6** CSV: 取込はスキーマ検証・型チェック後に投入。書き出しはフォーミュラインジェクション対策。
- **SEC-7** クライアント側チェックは UX 用。強制力は RLS ＋ サーバー側 DAL。
- **SEC-8** Supabase API キーは新形式（`sb_publishable_*`/`sb_secret_*`）へ移行中のため、実装時に最新の鍵運用を公式ドキュメントで確認。
- **SEC-9** 名刺画像は**非公開（private）Storageバケット**に保存。直接URLでの公開はしない。閲覧は**有効期限付き署名URL**を都度発行。Storage にも RLS/アクセスポリシーを設定し、`authenticated` のみアクセス可・`anon` 不可とする。署名URLは短い有効期限を設定し、期限切れ時はサーバー経由で再発行する（G-15）。
- **SEC-10** OCRに外部サービスを使う場合、名刺の個人情報を**どこに送信し・どこに保管するか**を事前審査する（国内保管 C-7 と無料維持 C-1 に反しないこと）。外部送信を避ける方式（ブラウザ内 or 自前/CF内処理 or 手元Claude）を優先検討（§6.5.1 / §12）。
- **SEC-11**（G-13）UI出し分けだけに頼らない**多層防御**。adminページ/機能は UI で非表示にしたうえで、**ルートガード＋サーバー側DAL**でも到達制御する（最終的な強制力は RLS）。
- **SEC-12** 企業の資料ファイル（§6.11 `company_documents`）も名刺と同じ**非公開バケット＋署名URL**で扱う（SEC-9 を適用）。アップロードは**拡張子・MIME・サイズの許可リスト検証**を通す（実行ファイル等を弾く）。Storage にも RLS を設定し `authenticated` のみ・`anon` 不可。署名URLは短い有効期限で都度発行する。全社横断の資料検索（§6.11.1）でも MCP には**メタデータのみ**返し、ファイル本体・署名URLは渡さない（SEC-X4）。
- **SEC-13** 活動履歴（§6.12 `activities`）も RLS で `authenticated` のみ・`anon` 不可。自動記録の `title`/`body` には個人名・商談内容など機微情報が入りうるため**短い要約に留め**、実体（議事録全文・名刺・資料）は元テーブル＋署名URL側に置き本表へ重複保存しない。MCP へ返す際は PII 最小化（SEC-X4）。監査用途のため活動行の物理削除は admin の例外操作とする（NFR-9）。

## 11. 受け入れ基準（MVP抜粋）

- ログインしないと顧客データに到達できない（URL直叩きでもRLSで弾かれる）。
- 企業の登録→一覧表示→絞り込み→編集→論理削除が一通り動く（論理削除した企業・配下担当者は既定で非表示）。
- 企業配下に担当者を追加でき、担当者に**名刺画像（表/裏）を保存・閲覧**できる。
- 名刺画像は**署名URLでのみ**閲覧でき、非公開バケットの直URLでは取得できない。
- needs/offers をマスタから選択でき、無いタグはその場で追加できる。
- 正常CSV（企業単位）を取込でき、列ずれ/型不正CSVはエラー行が提示され投入されない。
- 書き出しCSVで `=` 等で始まるセルがエスケープされている。
- ブラウザに `service_role`/secret キーが露出していない（ビルド成果物・ネットワークを確認）。
- （P1.5）名刺アップロード→OCR→確認・補正→企業＋担当者作成が動く（OCRエンジン決定後）。
- 論理削除した企業/担当者をゴミ箱で確認し、復元できる（G-03）。
- ログアウトでき、自分のパスワードを変更できる（G-01）。
- 非adminは管理画面に到達できない（UI非表示＋ルート/DALガード。G-13）。
